Project Architecture Overview

Part 1: General Project Structure

Based on the exploration of the codebase, here is a generalized overview of how the project is structured and how its components interact. The project is organized as a Monorepo, meaning all the different parts of the system live in a single repository and share dependencies.

It is divided into two main areas: User-Facing Applications and Shared Core Modules.

1. User-Facing Applications

These are the deployable entry points of the platform:

  • The Backend Gateway (API): This is the central nervous system of the platform. It handles all incoming requests, exposing endpoints for everything from user onboarding and billing to policy management, incident tracking, and health checks. It acts as the bridge between the web interfaces and the core business logic.
  • The User Dashboard: A modern, server-rendered web application where users (like CISOs or admins) log in to view analytics, configure security rules, manage access, and monitor the system's overall health.
  • Public Sites: Separate, lighter applications handle the public marketing pages and the developer documentation.

2. Shared Core Modules

Instead of putting all the logic into the API, the system's functionality is broken down into specialized, reusable modules. This keeps the code clean and allows different apps to use the exact same logic safely:

  • The Security & Policy Engine: This is the core engine of the platform. It contains a suite of detectors, parsers, and rules designed to scan and evaluate AI system behavior, ensuring everything complies with set policies.
  • The Interception Proxy: A critical module that sits between users and AI models. It acts as a gatekeeper, intercepting requests to enforce policies in real-time. It handles tasks like detecting anomalies, estimating token usage, enforcing rate limits, and parsing the communication protocols.
  • The Data Layer: A dedicated module responsible for all database interactions. It defines the structure of the data (schemas) and handles migrations, ensuring that data like audit logs, user configurations, and policies are stored and retrieved efficiently, often with multi-tenant separation.
  • Identity & Access Management: This module centralizes all security around who can access what. It manages user sessions for the dashboard, validates API keys for programmatic access, and enforces role-based permissions across the platform.
  • Compliance & Observability: Specialized modules exist to take the raw data and turn it into actionable insights. One focuses on generating formal compliance documentation (like for regulatory standards), while another aggregates metrics for dashboard reporting and alerting.

How Everything Flows Together

  1. A user or system makes a request (either to manage settings via the Dashboard or to route an AI prompt via the Interception Proxy).
  2. The Identity & Access Management module first verifies if the requester is authorized.
  3. If it's an AI request, the Proxy intercepts it and hands it to the Security & Policy Engine to scan for violations or anomalies.
  4. The results, along with general usage data, are recorded securely via the Data Layer.
  5. Finally, this stored data is processed by the Compliance & Observability modules so that administrators can view real-time reports and alerts back on the Dashboard.

In short, the architecture is designed to be a robust, modular pipeline where traffic is authenticated, rigorously analyzed against security rules in real-time, and deeply logged for reporting.

Part 2: API Architecture & Authentication Flow

General API Architecture

The backend API is built using Hono running on Bun, which provides an extremely fast, lightweight web server. It serves a dual purpose: it acts as a traditional REST API for the dashboard and as a high-performance interception Gateway for AI traffic.

  • Dual-Gateway System: The API essentially splits traffic into two main streams:
    • The Management REST API (/api/v1/*): This serves the Next.js web dashboard. It has heavily structured routes separated by domain (e.g., policies, alerts, audit-events, billing, observability).
    • The Runtime Gateways (/v1/mcp/* and LLM endpoints): These are dedicated, high-throughput pathways that intercept programmatic AI traffic. When an AI tool call or LLM prompt comes through, the gateway dynamically authenticates the key, loads the specific organization's policy rules from a cache, scans the payload for anomalies, logs the audit event, and proxies the traffic.
  • Multi-Tenant Data Isolation (Schema per Tenant): A critical architectural decision is how data is isolated. The API uses a custom middleware (schemaMiddleware) that intercepts requests, identifies the organization making the request, and points the database connection (Drizzle ORM) to a dedicated, secure Postgres schema specifically for that tenant. This ensures zero data bleed between customers.
  • Real-Time Event System: The API is heavily event-driven. Instead of blocking requests, tasks like webhook dispatching, alerting, and telemetry logging happen asynchronously. It also emits Server-Sent Events (SSE) so the web dashboard updates in real-time when policy violations or anomalies are detected.
  • Global & Gateway Rate Limiting: The API employs layered rate limiting backed by Redis. General API routes have a standard global limit, while the Runtime Gateways have aggressive, distinct limits tied specifically to the incoming API key's hashed identity to prevent abuse.

Authentication & Authorization Flow

The platform centralizes identity management using BetterAuth, extended with several custom plugins and middleware to support its enterprise requirements.

  • Human Users (Dashboard Auth):
    • Flow: Users log in via Email/Password or OAuth (Google, GitHub).
    • Session Management: The system issues secure, HTTP-only cookies. When a user requests a dashboard route, a sessionMiddleware intercepts the request, validates the cookie against the database (or Redis cache), and extracts the user's active Organization ID.
    • Enterprise Gates: There is middleware specifically designed to check the organization's billing plan. If a user tries to access advanced security features (like enabling MFA or SAML SSO), the API checks if their tier supports it before allowing the action.
  • Machine/Programmatic Users (Gateway Auth):
    • Flow: For traffic hitting the AI proxy gateways or data ingestion endpoints, session cookies aren't used. Instead, programmatic access relies on API Keys sent as Bearer tokens.
    • Security: To prevent sensitive keys from leaking in logs or being compromised, the gateway intercepts the raw Bearer token and immediately creates a SHA-256 hash. It then looks up this hash in the database to identify the organization.
    • Fallback: The main REST API (/api/v1/*) is smart enough to fall back to this API Key authentication if a dashboard session cookie isn't present, allowing SDKs to interact with the management API.
  • Organization Lifecycle Hooks: BetterAuth is heavily integrated into the organization lifecycle.
    • Provisioning: The moment a new organization is created during signup, a database hook automatically fires to execute migrations and provision a brand-new, isolated Postgres schema for them.
    • RBAC (Role-Based Access Control): When inviting new users to an organization, hooks intercept the invite to ensure the organization's tier supports advanced roles (like assigning an 'admin' vs 'viewer') and that they haven't exceeded their active user limits.
